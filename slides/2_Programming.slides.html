The scientific computing ecosystem

A very brief introduction for psychologists

A programmer and a psychologist walk into a bar...

  • ...actually, this rarely happens
  • Relatively few psychologists program
  • Some psychologists go into psychology because you don't need technical skills
  • Many psychologists think they don't need to program to be good researchers
  • Many psychologists think it's just not for them
  • It's true that you don't need to learn to program
  • You also don't need to know multiple regression
  • But the gap between what you can do with vs. without these skills is enormous (and growing)

  • "Split-apply-combine": a common strategy for analyzing data (Wickham, 2011)
  • Divide the data into groups (in this case, by country)
  • Operate on each piece separately
  • Combine the result into one data structure
  • Can be done very elegantly and quickly in R, pandas, etc.

    import seaborn as sns
import matplotlib.pyplot as plt

# Select valid rows
d = data[data['sex'].isin(['f','m']) & ~np.isnan(data['Stimuli']) & data['sexuality'].isin([1,2])]

# Rename sexuality and stimuli levels to be more descriptive
d.loc[:,'sexuality'] = d['sexuality'].replace({1.0: 'Heterosexual', 2.0: 'Homosexual'})
d.loc[:,'Stimuli'] = d['Stimuli'].replace({1.0: 'GayMen', 2.0: 'Lesbians'})

# Keep only countries with > 1,500 subjects
d = d.groupby('citizenship').filter(lambda x: x['hour'].count() > 1500)

# Order countries by ascending overall IAT effect
order = d.groupby('citizenship')['IAT_overall'].mean().order().index

# Plot with seaborn
sns.factorplot('citizenship', 'IAT_overall', data=d, row='sexuality', col='Stimuli', hue='sex', 
               join=False, hline=0, size=3, aspect=3, x_order=order, kind='point')

    • Should you spend the time?

  • Like many things in life: a tradeoff between short-term vs. long-term benefits
  • It takes some time to learn to program
  • During that time, you're less productive--and you may get very frustrated
  • Afterwards, the world is your oyster

    • What should you learn?

  • There are like a million programming languages out there
  • That's only a slight exaggeration
  • Many possible ways to classify and organize languages
  • Compiled vs. interpreted, statically vs. dynamically typed, functional vs. object-oriented vs. procedural, etc. etc.
  • Probably the most salient dimension for scientists: development vs. execution speed
  • Your time versus your machine's time: what's the bottleneck?
  • So, very crudely, we have: C/C++/Java vs. Python/Ruby/Perl
  • Newer languages (e.g., Julia, Go) may eventually give us the best of both worlds
  • Can also achieve good high-level performance via bridges to other languages
  • E.g., much of numerical analysis in Matlab, Python, R is done in C or FORTRAN

    So what should you use?

  • Depends on what you do
  • Common use cases for psychologists:
  • Text processing and general purpose scripting
  • Stimulus presentation/data collection
  • Statistical and numerical analysis
  • Visualization
  • Online experiments/web development

    • General-purpose scripting

  • You want a high-level language that does a lot of different things quickly
  • Code is typically interpreted, not compiled
  • Weak variable typing
  • (Relatively) easy to learn
  • Hides a lot of low-level stuff from programmer
  • Common languages: Python, Ruby, Perl, etc.
  • Choose based on ease of use, community, available libraries

    • Experimentation frameworks

  • Traditionally, psychologists use standalone packages
  • E-Prime, Presentation, etc.
  • These are easy to use, but not very flexible/powerful
  • Good for simple experiments
  • Force you to do more complicated things in very convoluted ways
  • E.g., E-Basic is an ineffably terrible 'language'
  • Recent push towards packages based on existing languages
  • E.g., PsychoPy in Python, Psychtoolbox in Matlab
  • Best of both worlds: simplifies typical use cases while providing flexibility of a full-featured language
  • (Potentially) open source, so you can contribute to development

    • Statistical/numerical computing and visualization

  • Most psychologists work predominantly in:
  • SPSS, R, Matlab/Octave, Python/NumPy, or SAS
  • SPSS is by far the most common, but is inflexible and has very poor programmatic support
  • SAS use is decreasing rapidly, but still used for large datasets
  • Matlab is fast, handles large datasets, and has solid plotting capabilities
  • But it's expensive, the language is clunky, and ecosystem is focused almost exclusively on numerical computing
  • R is free and has by far the most extensive support for statistical analysis
  • But it has a steep learning curve, clunky syntax, and chokes on large datasets
  • Python (w/ numpy and pandas) is fast, free, and has by far the largest ecosystem
  • Lacks analogs to many statistical packages implemented in R

    • Web and mobile development

  • Online/mobile data collection is a sensible thing to do
  • For simple surveys, you can use tools like SurveyMonkey, Qualtrics, etc.
  • More complex studies require some web or mobile app development
  • Server-side versus client-side programming
  • On the server, you can use whatever you want (Python, Java, etc.) to generate content
  • In the browser, you're stuck with JavaScript (for now)
  • In native mobile apps, you're tied to Java (Android) or Objective-C (Apple)
  • Web development ecoystem moves unbelievably quickly
  • Alternative syntaxes (e.g., CoffeeScript) make JavaScript more pleasant
  • Frameworks like jQuery do much of the work for you
  • Staying current is almost a full-time job

    • So, what should you use?

  • Again, depends on what you do
  • In an ideal world, you'd use the best tool for each job
  • In the real world, nobody has time for that!
  • If you only do one kind of thing, pick the best tool, e.g.,
  • e.g., R for comprehensive statistical analysis
  • C/C++ if performance is your sole consideration
  • For the web, JavaScript is unavoidable
  • If you want to do (almost) everything in one environment, Python wins by a landslide
  • Arguably the dominant general-purpose language in scientific computing at the moment
  • Popularity varies by field: R in bioinformatics, Matlab in neuroimaging, etc.
  • Marginal cost of learning a new language decreases with experience

    • The scientist as a software developer

  • Scientists:
  • Usually work alone on single-purpose analysis scripts
  • Just want the damn thing to work NOW
  • Don't care about code reuse or clarity
  • Comments? What are comments?
  • Professional developers:
  • Often work in teams on large projects
  • Need to write clear, manageable code
  • Expect much/most code to be re-used at some point
  • Place a premium on documentation and testing

    • Think like a developer!

  • Developers emphasize a lot of practices that seem onerous
  • Complex development tools
  • Comprehensive documentation and commenting
  • Modularity/code re-use
  • Automated testing
  • Version control

  • These may seem unnecessary for scientific software, but...
  • "Never again!" turns into "where did I put that..." surprisingly often
  • Small projects tend to mushroom
  • You don't realize how much easier your life could be
  • Not that onerous once it becomes habit

    • Development tools

  • A good IDE (integrated development environment) can boost productivity considerably
  • Manages your code, completes variable/method names, etc.
  • Eclipse, NetBeans, Xcode, Visual Studio, etc.
  • For small scientific projects, a full-blown IDE may be overkill
  • Lightweight text editors can do the job
  • Offer the key features (syntax highlighting, in-line execution, etc.) without overhead or learning curve
  • Sublime Text, TextMate, Notepad++, emacs, vi, etc.
  • The IPython Notebook provides an excellent development/testing/interactive environment
  • Not just for Python!
  • IPython has support for many other languages

    Document your code

  • Comment everything
  • Would I understand this code if I came back a year later?
  • Would anyone else understand it right now?
  • Tell the user why the code is doing what it's doing
  • There are no good reasons not to comment
  • If you want people to use your tools:
  • Make them as easy as possible to use
  • Write good documentation--especially tutorials
  • If no one else uses your code, comment it anyway
  • Will save you time later
  • If you're not doing things programmatically, you should still document your work

    • Modularize and abstract

  • Don't Repeat Yourself
  • Good programming is lazy programming
  • If you find yourself writing the same code repeatedly, abstract it
  • Encapsulate the functionality in a method or class
  • Write modular, generalizable code
  • Planning to write code to parse E-Prime log files?
  • Why not make it parse log files in general, and make E-Prime a special case?
  • Abstraction makes life better. For example:
  • All those import statements in Python let you do more in less code
  • CoffeeScript compiles down to much more verbose JavaScript
  • The E-Studio GUI is just a tool for generating (hideous) E-Basic code
  • Python/Numpy/Matlab rely heavily on C bindings, but you never see them

    • Version control

  • Systematic tracking of changes to your code (or data, analysis, papers, whatever)
  • Most common model is to check code in/out of a centralized repository

  • Many platforms
  • git, Mercurial, SVN, etc.
  • All have similar basic functionality: create repos, check out code, check in code, merge branches, etc.
  • Openly hosted version control platforms (GitHub, Bitbucket, etc.)
  • Increasingly popular way to both manage software and encourage collaboration and adoption
  • GitHub is by far the most popular platform
  • Just added new features for scientists, including doi assignment
  • Further reading

    • Why use version control?

  • You're collaborating on a project with other people
  • You occasionally make mistakes and need to roll back
  • You sometimes need to simultaneously have different versions of the same project
  • You have a hard time remembering where you put things
  • You want to have an easy way to annotate all aspects of your project
  • You like to be able to chart your progress
  • You want other people to use your stuff
  • You don't want to deal with people bugging you for data/code
  • ...and many other reasons

    • Testing

  • How do you know if your code works?
  • You don't
  • Unless you test it
  • New features tend to break old features
  • Automate everything!
  • A comprehensive set of tests allows you to find bugs before they affect anyone
  • Many developers swear by test-driven development
  • Write tests first, then write code to pass tests
  • Writing tests takes a lot of time
  • But debugging code blindly takes much more time
  • You're going to test anyway; may as well do it without the frustration
  • There are a few exceptions...

    • Google more, write less

  • Most of the problems you'll run into have already been raised and solved by other people
  • Use other people's solutions when you see them
  • Google everything!
  • I googled half of the things in these slides
  • Much of the trick to learning this stuff is knowing enough to be able to generate the right Google or Stack Overflow queries
